Product Requirement Documentation From First Principles
Abigail & Jason


CORE IDEA
Big satellites today are like old computers that filled entire rooms.
* Huge
* Expensive
* Built for one job
What happened to computers:
* They became smaller
* Cheaper
* More powerful
* One device now does many things
That shift has not fully happened in satellites yet.
________________


WHAT WE ARE DOING
We are applying that same shift to space:
Make small satellites do the work of multiple large satellites
________________


THE PROBLEM
Today:
* One satellite = one job
   * Earth imaging → one satellite
   * Climate monitoring → another
   * Communication → another
This leads to:
* High cost
* Long timelines
* Fragmented systems
Only large governments and companies can afford this.
________________


OUR APPROACH
Instead of:
One satellite = one function
We move toward:
One small satellite = multiple functions
Example (simple):
One satellite can:
* Take images
* Monitor land and crops
* Support basic communication
Not perfect at everything.
But useful across multiple things.
________________


WHY THIS MATTERS
If this works:
* Fewer satellites needed
* Cost drops significantly
* Smaller teams can build
* Africa can own infrastructure
________________


CURRENT REALITY
We are not building the full multipurpose satellite yet.
We are starting with:
One satellite
One use case (earth observation)
________________


WHY SOFTWARE FIRST
We begin with software because:
* Hardware is expensive
* Software is fast to build
* Simulation lets us test ideas safely
So first:
Design → Simulate → Validate → Then build
________________


CURRENT STEP (PHASE 0)
We are designing:
* A 3U CubeSat
* Earth observation payload (camera)
* Nadir-facing (pointing to Earth)
* Basic systems (power, comms, ADCS)
Goal:
Create something real enough to simulate
________________


TOOLS (WHAT WE ARE USING RIGHT NOW)
* Blender → for external structure and visualization
* Autodesk Fusion → for basic mechanical layout
What we are doing:
* Designing outer shell
* Placing major components (camera, panels, body)
What we are NOT doing yet:
* Full internal circuit design
* High-fidelity hardware modeling
* PCB-level architecture
Reason:
* Those tools require capital and high complexity
* Real engineering expertise we do not have yet
* We only need a physically believable model
________________


WHAT THE SOFTWARE WILL DO? (CORE FUNCTIONS)
The software we are building will:
1. Track and visualize satellites in orbit
   * Know where every satellite is in real time
2. Simulate satellite behavior
   * Orbit movement
   * Power usage
   * Coverage
3. Coordinate satellites using RL (IkirereMesh)
   * Avoid collisions
   * Optimize coverage
   * Manage constellation behavior
________________

ADCS — SIMPLE EXPLANATION
ADCS = Attitude Determination and Control System
What it does:
 It makes sure the satellite is pointing the right way, at the right time.
________________


THE PROBLEM
In space, the satellite is not stable.
It will:
      * Tumble
      * Rotate
      * Drift off direction
Nothing holds it steady.
Without ADCS:
      * The camera won’t face Earth
      * The antenna won’t point to ground stations
      * The solar panels won’t face the Sun
The satellite becomes unusable.
________________


HOW ADCS WORKS
1. Determination (Sense orientation)
The satellite figures out how it is currently oriented using sensors:
      * Sun sensors → detect Sun direction
      * Magnetometer → detects Earth’s magnetic field
      * Gyroscope → measures rotation
This allows it to know:
“I am not pointing correctly”
________________


2. Control (Fix orientation)
The satellite corrects itself using:
      * Reaction wheels → spin to rotate the satellite
      * Magnetorquers → interact with Earth’s magnetic field to adjust direction
No propulsion needed.
________________


FOR OUR DESIGN (EARTH OBSERVATION)
We want:
Camera always pointing down to Earth
This is called:
Nadir pointing
So ADCS will:
      * Detect when the satellite drifts
      * Adjust orientation
      * Keep it stable continuously
________________


WHAT THIS MEANS FOR PHASE 0
      * The satellite will not naturally stay facing Earth
      * ADCS is required for imaging to work
In our design:
      * Camera = Earth-facing side
      * ADCS = internal system that keeps it aligned
We do not model internal ADCS hardware yet
But we must include it conceptually
________________


SIMPLIFIED VERSION (FOR NOW)
      * Basic ADCS only
      * No high precision
      * Goal: keep camera pointing down
This is enough for simulation.
________________


IMPORTANT DISTINCTION
      * Orbit = where the satellite is
      * ADCS = how it is oriented
They are different systems.
________________


IMAGING STRATEGY
The satellite will:
      * Capture images only when passing over Africa
      * Stay idle or conserve power outside target regions
Reason:
      * Saves power
      * Focuses on primary use case
      * Reduces unnecessary data collection
________________


DATA HANDLING STRATEGY
The satellite will:
      * Capture images when passing over Africa
      * Store images onboard (internal memory)
      * Transmit data only when in range of a ground station
________________


Reason:
      * Real-time transmission requires high power
      * Continuous connection is not guaranteed in LEO
      * Storage + delayed transmission is more reliable
      * Matches constraints of a 3U CubeSat
________________


COMMUNICATION STRATEGY
The satellite will:
      * Use RF communication (UHF/VHF or S-band)
      * Establish connection only during ground station passes
      * Downlink stored data in short communication windows
________________


Reason:
      * Ground contact only happens for a few minutes per orbit
      * Limited power requires short, efficient transmissions
      * Simpler system for first version
________________


POWER USAGE STRATEGY
The satellite will:
      * Generate power using body-mounted solar panels
      * Store energy in onboard batteries
      * Prioritize power usage based on operation phase
Operation priority:
      1. Survival (basic systems always on)
      2. ADCS (maintain orientation)
      3. Payload (camera only when needed)
      4. Communication (only during downlink windows)
________________


Reason:
      * Power is the most limited resource
      * Not all systems can run at once
      * Requires controlled scheduling
________________


OPERATIONAL FLOW (SIMPLIFIED)
The satellite will operate in cycles:
      1. Orbiting Earth
      2. Detect when over Africa
      3. Activate camera → capture images
      4. Store data onboard
      5. Continue orbit
      6. Detect ground station
      7. Transmit stored data
      8. Return to low-power state
________________


Reason:
      * Matches real CubeSat operation
      * Efficient use of limited resources
      * Easy to simulate in software
________________


SOFTWARE ROLE (GROUND SYSTEM)
The software will:
      * Track satellite position in orbit
      * Determine when satellite is over Africa
      * Simulate power usage and system states
      * Decide when to capture images
      * Decide when to transmit data
      * Coordinate behavior across satellites (later with RL)
________________


Reason:
      * Software controls decision-making
      * Enables testing before hardware exists
      * Foundation for IkirereMesh coordination
